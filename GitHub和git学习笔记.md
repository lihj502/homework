## 一、GitHub和git的区别

​		其实一开始我是不清楚GitHub和git有区别的，以为是一个东西，因此在一开始要新建个人仓库的时候我首先先着手的就是git的个人仓库，不过后来在提交的时候终于发现不太对劲，在查阅了资料之后才算有一点明白他们的不同点了。

**以下是我对GitHub和git区分的叙述:**

1. 首先，最直观的是GitHub和git的**性质**不同
   * GitHub是一个**网站**。
   * git则是一个**软件**。

2. 其次，他们的**操作地点**有着不同

   * GitHub就类似于现在很火的“云”，它目前给我的感受就是它是程序员们专属的网盘，它将程序员们创建的一些文件、代码等储存在遥远的云服务器上，它可以记录你对你曾经**上传的文件**的更改。

     （操作地点在云端）

   * git作用更多是管理你的文件，很多人把它比作“时光机”，它可以记录你对**本地文件**的操作历史，记录历史版本。

     （操作地点在本地）

在查阅、浏览资料的过程中，了解到其实GitHub和git并非毫无联系

## 三、GitHub学习笔记

### 3.1 GitHub初印象

​		首先正如上文在比对GitHub和git的差别中提到的那样，GitHub是一个面向开源以及私有软件项目的平台。在网上看到有人调侃Github是全球最大男性交友网站:joy:,这也恰恰说明GitHub是世界各地的程序员的聚集地。

​		GitHub作为一个开放的平台，它除了在上文已经谈到的托管文件及历史版本管理功能以外，最吸引各地程序员的地方就是它良好的分享交流氛围。

​		作为世界各位顶级开发者的聚集地，在这个平台上分享有许多优秀、著名的开源项目，我们可以在遵守各种不同的license的情况下，免费试用和下载这些开源项目，随着这个功能同时产生的是每一位程序员都可以对各种免费开源项目在遵守其license的情况下（如Apache许可证就需要在你修改了代码以后，在被修改的文件中说明）对此项目进行修改完善，使此项目的质量提升，当然了即使目前对项目还没有能力做出更改，在对这个项目的学习过程中同样也是对自己的一种提升。

## 四、git学习笔记

### 4.1 git初印象

git是一个免费、开源的分布式版本控制系统

它的是由工作区，暂存区以及本地库构成的。各区域之间分别通过“git add”、“git commit”完成交流。

对于它的结构我有一些自己不成熟的分析和猜测，将在对git的总结中阐述。

**分布式:** 是可以在自己的电脑建立一个完整的版本库，这样子就可以避免必须联网（集中式的特点）的限制，并且方便了个人对自己的版本库的修改。

**版本控制系统:** 版本控制系统是是一个可以记录并查阅对文件所作的改动，同时还可以返回（可以防止误删或误该一些内容）它所控制的文件的内容版本变化的一个系统。



### 4.2 学习git的过程以及过程中出现的问题和解决方法

1. **首先要分清Git Bash和Git GUI的区别:**

   * Git Bash是命令行界面
   * Git GUI是图形界面

2. **接下来完成对用户名以及邮箱的填写**

   这里的注意事项是用户名在未来上传文件到GitHub上的时候可以被看见，所以最好就是将用户名和GitHub中的用户名统一，这样以防出现难以识别那个是自己对GitHub上的文件进行了修改。
   
   如果填写完毕没有什么变化则为填写成功，如果出现一大堆文字则出现了问题。对于此种情况要返回最初填写的命令行，看它下面系统给出的error是什么（我一开始的问题是吧global写成了globle:joy:）

3. **创建一个git的本地库，对于创建的方法，我在学习过程中了解到两种途径:**

   * 可以直接通过windows的图形界面，直接打开将要创建文件的地方git bash here。

   * 也可以通过git bash的命令行界面来一步步cd直至进入将要创建文件的地方。

4. **对git本地库的初始化**

   创建的文件是隐藏的目录，既可以用git命令行界面用指令“ll -la”来查看，又可以在windows的图形界面勾选隐藏的项目来使它显示出来。
   
   但是注意不可以随意的删除或修改.git文件中的任何子文件和子目录，不然可能会造成错误。

5. **接下来进行添加一个文件用来练习。**

   **添加文件同样也有两种途径:**
   
   - 第一个是可以利用touch 某某某.某格式的文件，它本来是用于修改文件或者目录的时间属性，包括存取时间和更改时间。但是如果它发现个人库中搜索不到所提到的文件，它会自己新建一个新的空白文件，利用它的这一个性质可以用git命令行终端新建一个空白文件。
   
   - 第二个方法自然是Windows图形界面可以直接新建所需文件。

6. **将新建的文件从工作区提交的到暂存区最后提交的到本地库（这也是git的内部工作原理）**

   在使用命令过程中 git commit -m“。。。” 这里m是message的意思，它管理是后面双引号中的文字，用来说明为何提交这个文件，相当于一个注释。
   
   **注意:** 没有放在本地库的文件git不会对其进行管理，即使放在本地库里，没有真正通过命令提交到本地库中，git也不会进行管理。
   
   有一个需要注意的小细节，当新建了一个txt文件以后，不需要在重命名的时候在结尾加入xxx.txt，因为系统已经默认生成的文件有了.txt的后缀只是隐藏了而已。如果多写了.txt那么在控制此文件，将其提交的到暂存区的过程中就需要文件名为xxx.txt.txt了，不然就会有fatal: pathspec 'demo.txt' did not match any files的内容出现:joy_cat:

7. **再次新建一个文件来看，尝试运用命令查看工作区和暂存区的状态**

   可以通过git status更直观的了解git管理的文件从工作区提交到暂存区最终提交到本地库的过程。

8. **修改文件中的内容**

   为了更直接的了解每一步发生了什么，我采用每一步操作之后都采用git status来查看状态

9. **尝试用命令查看历史记录**

   如果一次性修改太多，那么就会出现查看历史记录会出现无法完全展示，需要到冒号那里用空格展示下一页，用“b”返回上一页，当历史记录展示完毕以后会出现(END)，之后如何点q则退出历史记录浏览界面。
   
   当然，可以通过从网上搜索[Git命令大全](https://www.jianshu.com/p/93318220cdce)找可以简约展示历史记录的命令（个人比较偏爱git reflog的命令）

10. **学习核心功能版本控制**

    在版本后退或者返回最新版本的时候，打开的txt界面自动关闭，再次打开就时变成了所要改变的样子。
    
    可以每做完一步之后通过看git reflog来观察指针HEAD的指向。
    
11. **学习删除文件的命令**

    注意，rm的删除只是把工作区的文件删除
    
    需要将删除操作同步到暂存区以及本地库中，只需通过add，以及commit的操作即可实现，如果想要查看，则依然运用git status查看。

12. **学习如何找回被删除的文件**

    其实虽然文件已经被删除，但是它的历史记录依然存在，找回被删除的文件实际上就是将被删除的文件的版本通过版本控制回退到未删除的版本即可。
    
    此时找回的不只有工作区的文件，同时还有暂存区以及本地库中的文件

13. **学习如何使用Git Bash将本地项目推送到远程github仓库**

    首先要记录远程库的地址在code里查看
    
    由于地址很长因此可以在git中通过命令给此地址起一个别名
    
    通过push将本地项目推送到远程github仓库，同时它自己自动在远程库创建了一个新的分支，查看需要进入此分支。



### 4.3 git使用的小技巧

* 按住“ctrl”，之后滚动鼠标滚轮，可以对git命令终端进行放大和缩小。
* git中的命令和Linux是一样的，因此在想进行某些操作却又不知道它对应的命令的时候可以搜索Linux的命令来使用。当然，网络上也有相应的git命令整理[Git命令大全](https://www.jianshu.com/p/93318220cdce)十分便捷。
* 在无意中发现，我可以通过在光标在命令行上时，可以用方向键上下来将之前的命令重新编辑并使用（有点不适应复制粘贴以后没敲回车指令都自己运行，不过还好发现上下键的好处了:smile:）。
* 在git中，选中一段代码就相当于复制了，可以下一步直接在所要粘贴的地方点击鼠标右键，选中paste。



### 4.4 对本阶段git的心得总结

对于在git初印象中提到的分析和思考，在下面做出以下的阐述,在接触了git以后，给我最大的冲击就是通过学习git，能够通过命令行的控制，并运用查看状态指令更加直观的了解如何恢复先前的文件版本，以及找回删除文件的过程。

在学习git的过程中，我下意识的将其命令行界面与Windows 的图形界面相对比，想到其实版本控制在Windows上同样存在并且我也经常使用。

首先就是无论是word或者excel或者其他软件上，对于写错的东西或者误删的东西都可以通过"ctrl+z"恢复，"ctrl+y"重返，这就类似git中的版本控制，只不过git更加 的直观可以看到每一次的修改并通过只指针的指向变化来重返版本，而windows只是机械的一步步重返版本。

比如在Windows中错误删除文件同样可以通过"ctrl+z"键找回，在文档编辑时就类似于提交到和暂存区而之后进行保存就类似于完成到本地库的传输。

不过由于只是初学git，还没有对其完全理解，还有很多未知的功能需要学习，我会在未来的使用中继续了解git知识。



### 4.5 git使用后记

我一直在找有没有一条命令可以直接更改指定的文件名，最终成功的找到了相关的命令git mv -v 文件名 改后文件名之后commit提交

之后我就在想，似乎不一定非要使用命令更改名字，我又尝试了手动更改文件名字，之后再add commit push，也可以完成更改

然后我又用log查看了git的版本，我就震惊了，我可以看到我曾经对GitHub仓库的所有操作

最后使用git push推送到GitHub就能够完成所有的更改了

如果我使用GitHub，确实可以看到我曾经的版本，但是我没有找到方法直接通过GitHub进行版本回滚，但是我可以找到曾经删除的文件，然后复制文件中所有的内容，再创建新的文档以此来恢复文件（手动版本回滚）

当我想要尝试git版本回滚的时候，我发现我的历史太长了而且很多没有提交更改信息（因此都不知道每一次做了什么样的修改，所以以后再提交的时候一定要写修改的信息），所以我通过结合GitHub和git来进行版本回退，从GitHub的历史中可以直接浏览每一次的历史版本是什么样的，之后找到想要更改到哪个版本号，然后使用git reset来操作

但是版本回退以后出现版本冲突，我的仓库版本因为版本回退低于GitHub，因此无法更改GitHub，这时我通过push –force来强制覆盖

当版本更改完毕以后我再进行对GitHub的版本查看的时候就没有了，此时我只能通过查看历史命令log来找版本号再进行版本的更改

**PS:**  log如果太多，那么在输入这个命令的时候就使用`-number`这里number的意思是将要查看几个历史版本，用想查看的历史版本的数值来代替number即可

**真正核心的版本控制还是在git中，GitHub只是能浏览曾经的版本但是并不能做到版本回退**
